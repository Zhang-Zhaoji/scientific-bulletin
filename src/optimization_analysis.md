# 批量查询优化分析

## 当前场景

- **每周文献量**: ~400篇
- **每篇查询作者数**: 前3 + 后3 = ~6人
- **预计总查询**: 400 × 6 = 2400人次
- **唯一作者比例**: 估计 60-70%（大牛常发多篇）
- **重复查询浪费**: ~720-960次

## 方案对比

### 方案1: 当前并发 + 缓存（已实现）
```python
# 第一次查询: Dong Dong (2秒)
# 第二次查询: Dong Dong [CACHED] (0秒)
优点: 简单、有效
缺点: 第一次还是串行
```

### 方案2: 预提取 + 批量查询
```python
# 1. 预提取所有唯一作者 (800人)
# 2. 批量查询 OpenAlex
# 3. 回填到所有论文

优点: 减少重复查询、降低连接开销
缺点: 
- OpenAlex 没有官方批量作者查询接口
- 需要维护复杂映射
- 代码复杂度上升
```

### 方案3: PubMed 批量获取单位（可行）
```python
# 当前: 每篇论文单独请求 PubMed
# 优化: 一次请求 100 个 PMID，批量获取所有单位信息

E-utilities 支持:
https://eutils.ncbi.nlm.nih.gov/entrez/eutils/efetch.fcgi?db=pubmed&id=PMID1,PMID2,PMID3&retmode=xml

优点: 大幅减少 PubMed 请求数 (400次 -> 4次)
```

## 推荐策略

### 短期（当前）
保持现有方案，因为:
1. 每周处理时间可接受（~10-15分钟）
2. 缓存已解决重复查询问题
3. 代码复杂度低，稳定可靠

## 结论

> 对方说得对，但在你们的场景下（每周400篇，缓存已存在），收益有限。

> **建议**: 保持现有方案，如果以后每周处理 >2000 篇，再考虑更激进的优化。
